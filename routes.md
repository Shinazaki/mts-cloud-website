# Документация API MTS Spacehack

Все эндпоинты по умолчанию имеют префикс `/api`.  
Аутентификация осуществляется посредством передачи JWT-токена в заголовке `Authorization: Bearer <token>`.
Некоторые методы требуют роль администратора (`admin`) или администратора корпорации (`corporation_admin`).

> **Замечание:** Для работы с платформой сначала необходимо зарегистрироваться (`/api/auth/register`), либо войти (`/api/auth/login`). Вы получите `access_token` в заголовках и `refresh_token` в защищенных cookies. Маршруты для работы с Proxmox используют этот токен для определения ваших прав доступа.

## 1. Auth (Авторизация и сессии)
Обеспечивает жизненный цикл пользовательских сессий, ротацию токенов и смену паролей.

- **`POST /api/auth/register`**  
  **Зачем нужен:** Зарегистрировать нового пользователя в системе. Возвращает токены.
- **`POST /api/auth/login`**  
  **Зачем нужен:** Вход в систему по username/password. Возвращает новую пару токенов.
- **`POST /api/auth/refresh`**  
  **Зачем нужен:** Если `access_token` (живёт 15 минут) истёк, отправьте запрос сюда (браузер сам прикрепит cookie `refresh_token`), и получите новый `access_token` в заголовке.
- **`POST /api/auth/change_password`**  
  **Зачем нужен:** Позволяет пользователю поменять свой пароль (нужно знать старый). Сбрасывает все активные сессии ради безопасности.
- **`DELETE /api/auth/logout`**  
  **Зачем нужен:** Завершить текущую сессию (удаляет конкретный текущий `refresh_token`).
- **`DELETE /api/auth/logout_all`**  
  **Зачем нужен:** Экстренное завершение всех сессий на всех устройствах.

## 2. Users (Управление пользователями)
Работа с профилями пользователей и ролевой моделью приложения.

- **`POST /api/users`** `[admin]`  
  **Зачем нужен:** Позволяет администратору вручную создать пользователя (например, служебный аккаунт).
- **`GET /api/users`** `[admin]`  
  **Зачем нужен:** Получить список всех пользователей системы.
- **`GET /api/users/:id`**  
  **Зачем нужен:** Получить данные конкретного пользователя по внутреннему ID.
- **`PATCH /api/users/update_info`**  
  **Зачем нужен:** Обновить свои личные данные (имя, фамилия, почта). Для подтверждения требуется текущий пароль.
- **`PATCH /api/users/:id/roles`** `[admin]`  
  **Зачем нужен:** Назначение системных ролей (`admin`, `user`, `corporation_admin`). При назначении сбрасывает все сессии пользователя, заставляя его перелогиниться, чтобы токен обновился гарантированно для всех устройств.

## 3. VPS (Виртуальные машины Proxmox)
Обеспечивает CRUD-цикл управления виртуальными машинами QEMU, а также возможность работы со снапшотами (снимками состояния).

### Управление и CRUD
- **`GET /api/vps/admin/proxmox-vms`** `[admin]`  
  **Зачем нужен:** Получение сырого списка всех виртуалок прямо с ноды Proxmox (для сверки с БД).
- **`GET /api/vps/admin/cluster-resources`** `[admin]`  
  **Зачем нужен:** Получение всех ресурсов кластера (VM, хранилища, ноды).
- **`POST /api/vps/create`**  
  **Зачем нужен:** Основной метод для пользователя. Клонирует базовый шаблон операционной системы, устанавливает CPU/RAM/Disk, настраивает Cloud-Init (юзер, пароль, сеть) и возвращает созданную машину, привязывая её к пользователю.
- **`GET /api/vps`** `[admin]`  
  **Зачем нужен:** Получить список всех машин из БД.
- **`GET /api/vps/my`**  
  **Зачем нужен:** Получить список машин, которые принадлежат ТОЛЬКО текущему авторизованному пользователю.
- **`GET /api/vps/:id`**  
  **Зачем нужен:** Получить детальную информацию о конкретной ВМ.
- **`PATCH /api/vps/update`**  
  **Зачем нужен:** Изменить конфигурацию существующей ВМ (например, увеличить количество ОЗУ или диска).
- **`DELETE /api/vps/delete`**  
  **Зачем нужен:** Удалить ВМ из-под Proxmox (очистка дисков) и стереть её из базы данных.
- **`GET /api/vps/:id/monitoring`**  
  **Зачем нужен:** Получить графики и RRD-метрики о потребляемых ресурсах (CPU, RAM, Сеть).

### Управление питанием (Power Management)
- **`POST /api/vps/start`** — Включить машину
- **`POST /api/vps/stop`** — Жёстко выключить машину (отрубить питание)
- **`POST /api/vps/restart`** — Перезагрузить машину
- **`POST /api/vps/shutdown`** — Мягко выключить через ACPI (запрос ОС на корректное завершение)

### Управление снапшотами
- **`GET /api/vps/:id/snapshots`** — Список доступных снимков машины
- **`POST /api/vps/snapshots/create`** — Сделать новый снимок системы (можно с дампом памяти или без)
- **`POST /api/vps/snapshots/rollback`** — Откатить ВМ в состояние конкретного снимка
- **`DELETE /api/vps/snapshots/delete`** — Удалить старый снимок (для экономии места)

## 4. LXC (Linux Контейнеры Proxmox)
Легковесные изолированные окружения поверх одного ядра Linux. В отличие от VPS (KVM), они работают быстрее, но поддерживают только Linux.

- **`GET /api/lxc`** `[admin]`  
  **Зачем нужен:** Список всех LXC контейнеров хоста.
- **`POST /api/lxc/create`**  
  **Зачем нужен:** Создаёт новый LXC контейнер из tar-архива шаблона (по умолчанию Alpine). Можно передать свой SSH-ключ (`sshPublicKeys`) для безопасного доступа.
- **`GET /api/lxc/:proxmoxId`**  
  **Зачем нужен:** Сводная информация и текущий статус (работает/остановлен) контейнера.
- **`GET /api/lxc/:proxmoxId/config`** / **`PATCH /api/lxc/:proxmoxId/config`**  
  **Зачем нужен:** Чтение и изменение выделенных лимитов и конфигурации.
- **`GET /api/lxc/:proxmoxId/monitoring`**  
  **Зачем нужен:** Графики RRD-метрик конкретного LXC контейнера.
- **`POST /api/lxc/:proxmoxId/start`**, **`stop`**, **`shutdown`**, **`reboot`**  
  **Зачем нужен:** Управление питанием контейнера (аналогично VPS).
- **`DELETE /api/lxc/:proxmoxId`**  
  **Зачем нужен:** Удаление контейнера и его тома данных.

## 5. Tickets (Система тикетов)
Подсистема обратной связи для создания обращений в поддержку.

- **`POST /api/tickets`**  
  **Зачем нужен:** Пользователь создаёт тикет, описывая проблему. Привязывается к ID пользователя.
- **`GET /api/tickets`** `[admin]`  
  **Зачем нужен:** Администратор смотрит все заявки пользователей.
- **`GET /api/tickets/:id`**  
  **Зачем нужен:** Просмотр конкретного тикета.

## 6. Corporations (Корпорации / B2B)
Функционал группировки людей. `corporation_admin` может создавать корпорации и добавлять туда других пользователей.

- **`POST /api/corporations`** `[corporation_admin]`  
  **Зачем нужен:** Создать юрлицо/организацию в системе.
- **`GET /api/corporations`** `[corporation_admin]`  
  **Зачем нужен:** Вывести список всех организаций.
- **`GET /api/corporations/:id`** `[corporation_admin]`  
  **Зачем нужен:** Детальная инфа (например, список участников).
- **`DELETE /api/corporations/:id`** `[corporation_admin]`  
  **Зачем нужен:** Удалить организацию.
- **`POST /api/corporations/:id/members/:userId`** `[corporation_admin]`  
  **Зачем нужен:** Пригласить/Добавить юзера системы в организацию.
- **`DELETE /api/corporations/:id/members/:userId`** `[corporation_admin]`  
  **Зачем нужен:** Исключить юзера из организации.

## 7. Pools (Пулы ресурсов Proxmox)
Пулы в Proxmox нужны для объединения виртуалок и хранилищ в логические группы, чтобы назначать права (ACL) не на каждую виртуалку по отдельности, а сразу на пул (группу).

- **`GET /api/pools`** / **`GET /api/pools/:poolid`** `[admin]`  
  **Зачем нужен:** Чтение списка и деталей конкретного пула ресурсов.
- **`POST /api/pools`** `[admin]`  
  **Зачем нужен:** Создать новый пустой пул.
- **`PUT /api/pools/:poolid`** `[admin]`  
  **Зачем нужен:** Добавить (`vms=100,101`, `storage=local`) или удалить ВМ/хранилища из пула.
- **`DELETE /api/pools/:poolid`** `[admin]`  
  **Зачем нужен:** Уничтожить пул (сами машины при этом не удалятся, только группировка).

## 8. SDN (Software-Defined Network)
Управление программно-определяемыми сетями (VLAN, VXLAN, EVPN) для обеспечения изоляции трафика между клиентами.

- **`GET /api/sdn/zones`** / **`:zone`** `[admin]`  
  **Зачем нужен:** Зоны — это тип сетевой абстракции (VLAN, QinQ, VXLAN).
- **`GET /api/sdn/vnets`** / **`:vnet`** `[admin]`  
  **Зачем нужен:** VNet (виртуальная сеть) — логический свитч, привязанный к зоне SDN. К VNet подключаются сетевые интерфейсы (мосты) виртуальных машин.
- **`GET /api/sdn/vnets/:vnet/subnets`** `[admin]`  
  **Зачем нужен:** Подсети (например, 10.0.0.0/24), ассоциированные с этой VNet, используются для IPAM (раздачи IP адресов).
- **`POST /api/sdn/apply`** `[admin]`  
  **Зачем нужен:** ОЧЕНЬ ВАЖНЫЙ РОУТ. При изменении SDN в Proxmox конфигурация просто записывается в файл как черновик. Чтобы она начала маршрутизировать трафик на реальных линукс-бриджах, нужно нажать "Apply" — этот метод как раз применяет сетевые изменения на всю инфраструктуру.

## 9. Storage (Хранилища)
- **`GET /api/storage`** `[admin]`  
  **Зачем нужен:** Все сконфигурированные стораджи в кластере Proxmox (`local`, `local-lvm`, `nfs` и т.д.).
- **`GET /api/storage/node`** `[admin]`  
  **Зачем нужен:** Стораджи, подключенные к конкретной ноде, с их заполненностью (used/total).
- **`GET /api/storage/:storage/content`** `[admin]`  
  **Зачем нужен:** Через параметр `?type=iso` можно получить список загруженных образов ОС (ISO). Через `?type=vztmpl` можно получить список шаблонов LXC.

## 10. RBAC (Ролевая модель PVE - Roles, PVE Users, ACL)
Роуты для прямого контроля доступов **в самом Proxmox** (например, чтобы выдать клиенту ограниченный доступ прямо в веб-интерфейс Proxmox VE).

- **Roles (`/api/rbac/roles`)**  
  **Зачем нужен:** Создание и изменение кастомных ролей Proxmox (набор галочек, например `VM.Audit`, `VM.Console`).
- **PVE Users (`/api/rbac/users`)**  
  **Зачем нужен:** Создание пользователей в `pve` realm (для авторизации в Proxmox GUI).
- **ACL (`/api/rbac/acl`)**  
  **Зачем нужен:** Связывает Путь (Path) + Роль (Role) + Юзера (User).  
  *Пример сценария: Вы создали ВМ в пуле `/pool/client_A`, создали юзера `client@pve`. С помощью роута ACL вы выдаете роль `PVEVMUser` для юзера `client@pve` на путь `/pool/client_A`. И теперь он из-под своего аккаунта может перезагружать только свою виртуалку.*
